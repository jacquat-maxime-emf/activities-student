# Exercice de rappel — Scrum
**Sujet** (module **324**) : EPIC / User Stories / Definition Of Done (réactivation)

**Resources** : Les présentations du module 306

**Modalité** : Par groupe de 2 (ou 3)

La méthodologie SCRUM a déjà été abordé en détail lors du module 306. Il s'agit ici de faire un rappel, en particulier les notions de User Story et de Definition Of Done.

---
## Objectifs
À l’issue de l’exercice, les apprenants seront capables de :
1. Formuler des User Stories correctes en respectant la structure standard et en mettant en évidence la valeur métier.
1. Différencier besoin fonctionnel et solution technique, en identifiant les erreurs courantes dans les User Stories.
1. Définir des critères d’acceptation clairs et testables pour valider une User Story.
1. Comprendre le rôle et l’utilité de la Definition of Done dans un cadre Scrum.
1. Construire une Definition of Done partagée servant de référence commune à l’équipe.
1. Évaluer l’état “Done” d’une User Story en s’appuyant sur des règles explicites plutôt que sur une perception individuelle.

---
## Contexte
Vous travaillez en équipe Scrum sur une application web de gestion de bibliothèque.

Par groupe de 3 réalisez une présentation PowerPoint qui reprend les livrables attendus dans les parties ci-dessous.

---
## Partie 1 – Identification des EPICs
### Consigne
À partir du contexte global, identifiez 3 EPICs minimum pour l’application.

### Règles
Rappel, un EPIC doit :
- représenter un objectif fonctionnel large,
- être exprimé du point de vue du produit, pas de la technique,
- pouvoir être découpé en plusieurs User Stories.

Exemple (indicatif) : Gestion du catalogue de livre

### Livrable attendu
Liste des EPICs avec 1 phrase descriptive par EPIC.

---
## Partie 2 – Déclinaison d’un EPIC en User Stories
### Consigne
Choisissez 1 EPIC et déclinez-le en 3 à 4 User Stories.

### Règles
Chaque User Story doit :
- respecter le format : **En tant que … je veux … afin de …**
- apporter une valeur métier claire,
- être compréhensible et testable.

Les critères d'acceptation doivent être rédiger avec la méthode **Gherkin** (Etant donné ... Quand je ... Alors ...).

### Livrable attendu
Pour chaque User Story :
- la formulation complète,
- 2 critères d’acceptation minimum.

---
## Partie 3 – Identifier les erreurs
### Consigne
Les éléments suivants sont proposés pour l’EPIC Gestion du catalogue.
Indiquez s’il s’agit d’un EPIC, d’une User Story valide ou d’un élément incorrect, et justifiez brièvement.
1. « Créer la table books en base de données »
1. « En tant que bibliothécaire, je veux ajouter un livre afin d’enrichir le catalogue »
1. « Créer une fonctionnalité de suppression de livre »
1. « En tant qu’utilisateur, je veux une base de données performante »
1. « Améliorer la gestion des livres »

### Livrable attendu
Pour chaque éléments :
- L'indication s'il s'agit un EPIC, d’une User Story valide ou d’un élément incorrect.
- La justification dans le cas où il s'agit d'un élément incorrect.

---
## Partie 4 – Definition of Done
### Consigne
Définissez **UNE** Definition of Done commune applicable à toutes les User Stories du projet.

### Livrable attendu
Une checklist claire et partagée, des points à vérifier lorsqu'une User Story est considérée comme terminée.

---
## Partie 5 – Questions additionnelles 
Répondez aux questions en argumentant et en justifiant vos éléments. 

1. Une User Story respectant les critères d’acceptation mais pas la Definition of Done est-elle Done ?
1. La Definition of Done peut-elle varier d’une équipe à l’autre ?
1. Qui est responsable du respect de la Definition of Done ?

### Livrable attendu
Une réponse et sa justification pour chacune des définitions.
